# RC1 – Scenario Precision Fix

## Status
Completed and validated

## Problem

The ScenarioHeaderGenerator (SHG) limited numerical precision of core values:

- StartJD / StopJD truncated to 6 decimal places
- ScenarioID used inconsistent formatting
- Result:
  - loss of scientific precision
  - non-deterministic ScenarioIDs
  - potential instability in validation pipeline

## Root Cause

Fixed formatting:

v.ToString("0.000000")

This caused:
- hard truncation
- irreversible precision loss

## Solution

### 1. Core Precision

Core values now use:

v.ToString("G17")

Result:
- full double precision
- round-trip safe
- deterministic

### 2. ScenarioID Precision

JD values in ScenarioID:

- truncated (not rounded)
- fixed to 3 decimal places

Example:

Core:
2459845.640277778

ScenarioID:
2459845.640

### 3. Location Formatting

Location values (Lat, Lon, Elevation):

- fixed to 6 decimal places

Reason:
- deterministic formatting
- stable diffs
- improved readability

### 4. Hash Consistency

CoreCanonicalizer updated to use G17

Result:
- hash based on full precision
- no mismatch between JSON and canonical form

## Validation

### Test Scope

- TS-A (2 scenarios)
- TS-B (2 scenarios)

### Results

- Precision increased to ~17 digits
- ScenarioID correctly truncated
- TimeScale consistent (TDB)
- CoreHash changed as expected
- Run/LastRun binary identical
- Idempotency confirmed

### Observations

Floating point representation effects observed:

Example:
2443872.704000 → 2443872.7039999999

This is expected behavior of IEEE 754 double precision.

## Design Decision

SHG does NOT validate TimeScale.

Reason:
- Input will be generated by a controlled GUI tool
- SHG becomes a deterministic pipeline component
- Validation responsibility moves to input tooling

## Outcome

- Scientific precision restored
- Deterministic pipeline behavior achieved
- Architecture aligned with future ScenarioCreationTool
