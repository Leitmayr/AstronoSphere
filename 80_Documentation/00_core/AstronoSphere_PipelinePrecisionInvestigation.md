## (GroundTruth / AstronoTruth Validation Standard)

---

## 1. Purpose

This document defines the **precision and validation acceptance criteria** for GroundTruth datasets generated by AstronoTruth.

Goal:
- ensure deterministic behavior
- avoid false regression failures
- define physically meaningful tolerance thresholds
- establish a clear distinction between **numerical noise** and **real errors**

---

## 2. Fundamental Principle

> The **Core.Time** values are the single source of truth for all temporal definitions.

Rules:
- Time values MUST be read from `Core.Time`
- Time values MUST NOT be reconstructed from `ScenarioID`
- Time values MUST NOT be modified, rounded, or reformatted
- JSON precision MUST be preserved exactly (string-based propagation)

---

- Minimum required precision: **≥ 9 decimal places (JD)**

## 3. Precision Levels

### 3.1 Input Precision (Core)

- Minimum required precision: **≥ 9 decimal places (JD)**
- No enforced padding
- No truncation allowed
- No rounding allowed

Example: 2463504.088194444


---

### 3.2 Request Precision (Horizons)

- MUST match Core precision exactly
- MUST be passed as string (no double conversion)
- MUST be reproducible (CanonicalRequest)

---

### 3.3 Output Precision (Horizons Response)

- Determined by Horizons internally (IEEE754 / solver dependent)
- Typically:
  - slight floating-point variation
  - small deviations from input timestamp

---

## 4. Determinism Criteria (MANDATORY)

### 4.1 Internal Determinism
Run == LastRun → MUST be binary identical


This is the **primary validation criterion**.

---

### 4.2 Hash Stability

- RequestHash MUST be stable
- EpochHash MUST be stable
- CanonicalRequest MUST be identical

---

## 5. Numerical Tolerance (External Comparison)

Comparison against historical or external datasets:

### 5.1 Time Deviation Threshold

Acceptable deviation: 
Δt ≤ 0.001 seconds (1 ms)


Rationale:
- Horizons internal rounding
- floating-point representation
- solver precision

---

### 5.2 Expected Range

Typical observed deviation: Δt ≈ 0.0001 – 0.0005 seconds


(= sub-millisecond range)

---

### 5.3 Interpretation

| Deviation | Meaning |
|----------|--------|
| < 1 ms | acceptable numerical noise |
| 1–10 ms | investigate |
| > 10 ms | likely input inconsistency |
| > 100 ms | definite error |

---

## 6. Physical Interpretation

At heliocentric distances (~9 AU): 
Δt = 0.00024 s → ≈ 72 meters


Conclusion:

> Sub-millisecond deviations are **physically negligible** and MUST be accepted.

---

## 7. Regression Policy

### 7.1 Allowed Changes

- change of precision source (ScenarioID → Core)
- increased precision in Core
- corrected canonical request generation

---

### 7.2 Forbidden Changes

- loss of precision
- non-deterministic output
- inconsistent CanonicalRequest

---

### 7.3 Baseline Reset Rule

A baseline MUST be regenerated if:

- precision source changes
- canonicalization changes
- measurement definition changes

---

## 8. Key Insight

> Historical datasets may contain **higher apparent precision from incorrect sources**.

Example:
- ScenarioID provided 8 dp
- Core provided only 6 dp

Correct behavior:
Use Core (correct source)
→ regenerate baseline


---

## 9. Final Acceptance Rule

A dataset is considered **valid** if:

- Core.Time is used as input
- Run == LastRun (binary identical)
- CanonicalRequest is stable
- deviation vs historical data ≤ 1 ms

---

## 10. Summary

AstronoSphere M1.9 defines precision correctness as:

- **source correctness (Core)**
- **deterministic reproducibility**
- **physically negligible deviation tolerance**
